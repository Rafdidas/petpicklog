# 반려동물 생활 플랫폼 설계서 v0.4

## 1. 문서 목적

이 문서는 기존 **반려동물 용품 가격 비교·정보 서비스 3차 설계서 v0.3**의 방향을 재정리한 4차 설계서이다.

기존 서비스는 반려동물 용품 검색, 관심상품 저장, 저장 당시 가격 기록, 현재 가격 재조회, 가격 차액 확인, 동물병원 조회, 기본 리뷰 작성을 중심으로 구성되어 있었다.

v0.4에서는 서비스 방향을 단순 가격 비교 서비스가 아니라 **반려동물 생활 플랫폼**으로 변경한다.

다만 초기 MVP 범위는 넓히지 않고, 기존에 구현된 기능을 유지하면서 화면 구조와 서비스 포지셔닝을 재정리한다.

---

## 2. 서비스 방향 변경

### 기존 방향

```txt
반려동물 용품 가격 비교·정보 서비스
```

### 변경 방향

```txt
반려동물 생활 플랫폼
```

이 서비스는 단순히 반려동물 용품 가격만 비교하는 서비스가 아니라, 반려인이 자주 확인하는 용품 가격, 관심상품, 동물병원 정보, 반려생활 가이드를 한곳에서 확인할 수 있는 생활형 플랫폼을 목표로 한다.

```txt
사료, 간식, 배변용품 가격부터 우리 동네 동물병원 정보까지 한곳에서 확인하는 반려동물 생활 플랫폼
```

```txt
반려동물 용품 가격과 생활 정보를 함께 관리하는 반려생활 플랫폼
```

## 5. 서비스 포지셔닝

이 서비스는 다나와처럼 가격비교 흐름을 참고하되, 전체 정체성은 멍냥보감처럼 반려생활 플랫폼에 가깝게 가져간다.

다만 초기 MVP에서는 기능을 과도하게 넓히지 않고 아래 범위에 집중한다.

### 5.1 1차 핵심

```txt
용품 가격 확인
관심상품 저장
저장 당시 가격과 현재 가격 비교
```

### 5.2 1차 보조

```txt
동물병원 찾기
공공데이터 안내
```

### 5.3 2차 이후 확장

```txt
반려생활 가이드
급여량 계산기
알러지 메모
예방접종 메모
입양 정보
상품 후기 고도화
```

## 6. 현재 프로젝트 상태

현재 서비스는 1차 MVP 기준으로 핵심 기능의 골격이 구현된 상태이다.

구현된 주요 흐름은 다음과 같다.

```txt
상품 검색
→ 상품 후보 저장
→ 찜 저장
→ 저장 당시 가격 기록
→ 현재 가격 재조회
→ 가격 차액 확인
→ 동물병원 조회
→ 기본 리뷰 작성
```

기술 스택은 다음과 같다.

```txt
Next.js
TypeScript
SCSS
Supabase
Vercel
CSS Module 미사용
Tailwind CSS 미사용
```

현재 페이지 구조는 다음과 같다.

```txt
/           홈
/products   상품 검색
/saved      찜 목록
/hospitals  동물병원 조회
/auth       로그인 / 회원가입
/api/products/search  상품 검색 API
```

현재 DB는 1차 MVP 기준으로 아래 테이블이 구성되어 있다.

```txt
profiles
external_products
products
saved_products
price_histories
product_reviews
animal_hospitals
```

현재 구현된 기능은 다음과 같다.

```txt
네이버 쇼핑 API 상품 검색
API 키가 없을 경우 데모 데이터 사용
검색 결과를 external_products 후보로 저장
로그인 사용자 상품 찜
saved_products 저장
찜 당시 가격 saved_price 저장
price_histories 가격 기록 저장
찜 목록에서 현재 가격 재조회
가격 차액 표시
마지막 가격 확인 시각 표시
동물병원 데이터 조회
지역 / 병원명 / 주소 검색
전화번호 표시
네이버 지도 외부 링크
기본 상품 리뷰 작성
```

검증 상태는 다음과 같다.

```txt
npm run lint 통과
npm run build 통과
```

## 8. Header 메뉴 변경

### 8.1 기존 메뉴

```txt
용품 검색
저장한 상품
동물병원 찾기
로그인 / 로그아웃
```

### 8.2 변경 메뉴

```txt
가격 확인
관심상품
동물병원
가이드
로그인 / 로그아웃
```

### 8.3 메뉴명 변경 이유

`용품 검색`보다 `가격 확인`이 사용자의 목적에 더 가깝다.

`저장한 상품`보다 `관심상품`이 쇼핑 서비스와 플랫폼 서비스의 문맥에 더 자연스럽다.

`동물병원 찾기`는 메뉴에서는 짧게 `동물병원`으로 정리한다.

`가이드`는 추후 반려생활 콘텐츠와 계산기 기능을 확장하기 위한 진입점으로 사용한다.

# 9. 화면 설계

## 9.1 홈 `/`

### 화면 목적

홈은 단순 소개 페이지가 아니라, 반려생활에 필요한 주요 기능으로 진입하는 시작 화면이다.

사용자가 처음 들어왔을 때 아래 흐름을 자연스럽게 이해해야 한다.

```txt
1. 자주 사는 용품 가격을 확인한다.
2. 관심상품으로 저장한다.
3. 나중에 가격 변화를 다시 확인한다.
4. 필요하면 동물병원 정보와 반려생활 가이드를 확인한다.
```

# 16. 리뷰 위치 변경

## 16.1 기존

홈 대시보드에서 리뷰 작성 유도

## 16.2 변경

리뷰 작성 유도는 홈에서 제거한다.

리뷰는 아래 위치에서만 노출한다.

```txt
1. 상품 상세 페이지
2. 관심상품 페이지
3. 구매하러 가기 클릭 이후
4. 가격 확인을 여러 번 한 상품
```

## 16.3 이유

첫 방문자는 아직 상품 검색도 하지 않았고, 관심상품도 저장하지 않았기 때문에 리뷰 작성 유도가 어색하다.

리뷰는 사용자가 특정 상품에 관심을 가진 이후에 자연스럽게 유도한다.

# 19. 내부 상품 DB 승격 흐름

## 19.1 현재 상태

현재 `products` 테이블은 존재하지만 실제 서비스 흐름에서는 거의 사용하지 않는다.

현재는 외부 쇼핑 API 결과를 `external_products`로 저장하고, 사용자가 찜하거나 리뷰한 상품을 기준으로 데이터가 쌓이는 구조이다.

---

## 19.2 v0.4 정책

v0.4에서도 `external_products → products` 승격은 본격적으로 진행하지 않는다.

아직 상품 리뷰 수와 저장 수가 충분하지 않기 때문이다.

다만, 상품 상세 페이지와 가격 기록 구조는 나중에 내부 상품으로 승격해도 유지될 수 있게 설계한다.

---

## 19.3 추후 승격 기준

아래 조건 중 하나를 만족하면 내부 상품 DB로 승격할 수 있다.

```txt
찜 5회 이상
리뷰 3개 이상
검색 빈도 높음
관리자가 직접 지정
가격 추적 대상 지정
```

# 25. 결론

v0.4 설계의 핵심은 기능을 더 많이 추가하는 것이 아니라, 기존 기능의 방향성을 **반려동물 생활 플랫폼**으로 재정리하는 것이다.

기존 서비스는 상품 검색, 관심상품 저장, 가격 기록, 동물병원 조회, 기본 리뷰 작성까지 구현되어 있다.

v0.4에서는 이 기능들을 다음 흐름으로 재배치한다.

```txt
반려생활 홈에서 주요 기능을 확인한다.
자주 사는 용품 가격을 검색한다.
관심상품으로 저장한다.
저장한 상품의 가격 변화를 확인한다.
상품 상세에서 가격 기록과 후기를 확인한다.
필요하면 동물병원 정보를 조회한다.
반려생활 가이드로 확장한다.
```

이 구조로 가면 기존 1차 MVP를 버리지 않고, 현재 DB 구조를 유지하면서 서비스의 정체성을 단순 가격 검색 서비스에서 반려동물 생활 플랫폼으로 확장할 수 있다.
