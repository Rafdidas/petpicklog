# 반려동물 용품 가격 비교·정보 서비스 3차 설계서 v0.3

## 1. 현재 프로젝트 상태

현재 서비스는 1차 MVP 기준으로 핵심 기능의 골격이 구현된 상태이다.

구현된 주요 흐름은 다음과 같다.

```txt
상품 검색
→ 상품 후보 저장
→ 찜 저장
→ 저장 당시 가격 기록
→ 현재 가격 재조회
→ 가격 차액 확인
→ 동물병원 조회
→ 기본 리뷰 작성
```

기술 스택은 다음과 같다.

```txt
Next.js
TypeScript
SCSS
Supabase
Vercel
CSS Module 미사용
Tailwind CSS 미사용
```

현재 페이지 구조는 다음과 같다.

```txt
/           홈
/products   상품 검색
/saved      찜 목록
/hospitals  동물병원 조회
/auth       로그인 / 회원가입
/api/products/search  상품 검색 API
```

현재 DB는 1차 MVP 기준으로 아래 테이블이 구성되어 있다.

```txt
profiles
external_products
products
saved_products
price_histories
product_reviews
animal_hospitals
```

현재 구현된 기능은 다음과 같다.

```txt
네이버 쇼핑 API 상품 검색
API 키가 없을 경우 데모 데이터 사용
검색 결과를 external_products 후보로 저장
로그인 사용자 상품 찜
saved_products 저장
찜 당시 가격 saved_price 저장
price_histories 가격 기록 저장
찜 목록에서 현재 가격 재조회
가격 차액 표시
마지막 가격 확인 시각 표시
동물병원 데이터 조회
지역 / 병원명 / 주소 검색
전화번호 표시
네이버 지도 외부 링크
기본 상품 리뷰 작성
```

검증 상태는 다음과 같다.

```txt
npm run lint 통과
npm run build 통과
```

## 2. 2차 설계 목적

2차 설계의 목적은 새로운 기능을 무작정 추가하는 것이 아니라, 현재 구현된 기능을 사용자가 이해하기 쉬운 화면 구조로 재정리하는 것이다.

현재 서비스는 기능의 골격은 만들어졌지만, 화면상으로는 아직 아래 가치가 명확하게 드러나지 않는다.

```txt
이 서비스가 무엇을 해주는지
어떤 상품을 검색할 수 있는지
상품 가격을 어떻게 확인하는지
저장한 상품의 가격 변화를 어떻게 보는지
동물병원 정보는 어떤 역할인지
리뷰는 언제 작성하는지
```

따라서 2차에서는 아래 방향으로 화면을 재구성한다.

```txt
1. 홈을 대시보드형 시작 화면으로 재구성
2. 상품 검색 화면을 실제 검색 도구처럼 개선
3. 저장한 상품 화면을 가격 추적 중심으로 재구성
4. 상품 상세 페이지 추가 여부 결정
5. 가격 히스토리 표시 위치 정리
6. 리뷰는 상세 구조보다 후기 목록 표시부터 구현
7. 동물병원 화면은 공공데이터 기반 정보 페이지로 정리
```

---

## 3. 2차 핵심 방향

2차의 핵심 방향은 다음과 같다.

```txt
/products는 상품 검색 도구처럼
/saved는 가격 추적함처럼
/products/[id]는 상품 가격 기록과 후기의 중심 페이지로
/hospitals는 공공데이터 기반 정보 페이지로
/는 서비스의 핵심 기능을 연결하는 대시보드로
```

## 4. IA 재정리

2차 기준 페이지 구조는 다음과 같이 정리한다.

```txt
/
- 대시보드

/products
- 반려동물 용품 검색

/products/[id]
- 상품 상세
- 가격 기록
- 후기 목록
- 후기 작성

/saved
- 저장한 상품
- 가격 추적 요약
- 가격 변동 확인

/hospitals
- 동물병원 찾기
- 공공데이터 기반 병원 정보

/auth
- 로그인 / 회원가입
```

헤더 메뉴명은 다음처럼 단순하게 유지한다.

```txt
용품 검색
저장한 상품
동물병원 찾기
로그인 / 로그아웃
```

`가격 추적`이라는 표현은 화면 내부에서 사용하고, 메뉴명은 처음에는 `저장한 상품`으로 둔다.

이유는 일반 사용자가 처음 봤을 때 `가격 추적`보다 `저장한 상품`이 더 직관적이기 때문이다.

# 5. 화면 설계

## 5.1 대시보드 `/`

### 화면 목적

대시보드는 사용자가 서비스에 들어왔을 때, 서비스의 핵심 기능을 한 화면에서 이해하고 바로 사용할 수 있게 하는 시작 화면이다.

단순 소개 페이지가 아니라 아래 기능으로 진입하는 중심 화면 역할을 한다.

```txt
용품 검색
대표 상품 탐색
저장한 상품 가격 확인
최근 확인 상품
동물병원 조회
리뷰 작성 유도
공공데이터 안내
```

---

대시보드에 포함할 핵심 정보는 다음과 같다.

```txt
1. 용품 검색
2. 대표 상품 카드
3. 저장한 상품 가격 확인
4. 최근 가격 확인 상품
5. 동물병원 조회
6. 리뷰 작성 유도
7. 공공데이터 / 가격 정보 안내
```

---

## 5.1.2 대시보드 섹션 구성

대시보드는 아래 순서로 구성한다.

```txt
7. 데이터 출처 안내
```

```txt

우리 애 용품, 다시 찾기 쉽게 저장하고 가격을 확인하세요.
```

## 5.1.4 대표 상품 카드 영역

### 목적

사용자가 어떤 상품을 검색하고 저장할 수 있는지 직관적으로 보여준다.

단순 버튼만 있는 화면보다, 실제 상품 카드가 보여야 사용자가 가격 비교/저장 서비스라는 것을 빠르게 이해할 수 있다.

### 섹션명 후보

```txt
대표 용품 둘러보기
많이 찾는 반려동물 용품
오늘 확인한 반려동물 용품
가격 확인이 가능한 용품
```

1차 추천 섹션명:

```txt
대표 용품 둘러보기
```

### 대표 카테고리

```txt
사료
간식
배변용품
고양이 모래
샴푸 / 위생용품
영양제
장난감
하우스 / 이동장
```

### 상품 카드 표시 정보

```txt

상품명
카테고리
쇼핑몰명
현재 확인 가격
최근 확인 시각
찜하기
가격 확인하기
구매하러 가기
```

### 카드 예시

```txt
[사료]

```

```txt

네이버 쇼핑 기준
15,800원

[찜하기] [가격 확인]
```

```txt

24,500원
마지막 확인: 2026.05.18
[찜하기] [가격 확인]
```

### 데이터 기준

대표 상품 카드는 초기에는 아래 방식 중 하나로 구성한다.

```txt
1. 데모 상품 데이터
2. 최근 검색된 상품
3. 최근 가격 확인된 상품
4. 관리자가 지정한 대표 검색어 기반 상품
```

초기에는 데모 데이터와 실제 네이버 API 데이터를 혼합할 수 있다.

단, 데모 데이터와 실제 데이터는 UI에서 출처를 명확하게 표시한다.

```txt
데모 데이터
네이버 쇼핑 기준
최근 확인 가격
```

## 5.1.5 저장한 상품 가격 확인 영역

### 목적

로그인 사용자에게 저장한 상품의 가격 추적 상태를 요약해서 보여준다.

이 영역은 `/saved`로 이동시키는入口 역할을 한다.

### 로그인 사용자 표시 정보

```txt
저장한 상품 개수
가격이 내려간 상품 개수
가격이 오른 상품 개수
최근 가격 확인 시각
최근 가격 변동 상품
```

### 예시

```txt
저장한 상품 8개
가격 하락 2개
가격 상승 3개
최근 확인 2026.05.18
```

### 비로그인 사용자 표시 정보

```txt
관심 상품을 저장하면 가격 변화를 확인할 수 있어요.
로그인하고 가격 추적을 시작해보세요.
```

### CTA

```txt
저장한 상품 보기
로그인하고 저장하기
```

## 5.1.6 최근 가격 확인 상품 영역

### 목적

서비스가 실제로 가격을 확인하는 서비스라는 느낌을 준다.

`최저가`라는 표현은 아직 동일 상품 매칭이 완성되지 않았기 때문에 조심해서 사용한다.

### 추천 섹션명

```txt
최근 가격을 확인한 상품
```

### 표시 정보

```txt
상품 이미지
상품명
카테고리
쇼핑몰명
현재 확인 가격
마지막 확인 시각
가격 확인하기
```

### 문구 정책

사용 가능한 표현:

```txt
최근 확인 가격
현재 확인 가격
네이버 쇼핑 기준 가격
마지막 확인 시각
```

피해야 할 표현:

```txt
무조건 최저가
실시간 최저가
가장 저렴한 상품
전체 쇼핑몰 최저가
```

## 5.1.7 동물병원 조회 영역

### 목적

공공데이터 기반 동물병원 정보를 대시보드에서 자연스럽게 노출한다.

상품 검색 서비스에만 머물지 않고 반려동물 생활 정보 서비스로 확장할 수 있는 기반을 만든다.

### 표시 정보

```txt
지역 검색창
지역 선택
동물병원 2~3개 미리보기
병원명
주소
전화번호
영업상태
지도 링크
```

### CTA

```txt
우리 동네 동물병원 찾기
```

### 안내 문구

```txt
동물병원 정보는 공공데이터를 기반으로 제공됩니다.
방문 전 운영 여부와 진료 가능 여부를 병원에 직접 확인해주세요.
```

## 5.1.8 리뷰 작성 유도 영역

### 목적

리뷰를 서비스의 장기 자산으로 쌓기 위한 진입점을 만든다.

다만 2차에서는 리뷰 상세 구조보다, 작성된 후기를 보여주고 저장 상품과 연결하는 데 집중한다.

### 표시 조건

```txt
로그인 사용자
저장한 상품이 있음
아직 후기를 작성하지 않은 상품이 있음
```

### 문구 예시

```txt
사용해본 상품이 있다면 후기를 남겨주세요.
다른 보호자의 선택에 도움이 될 수 있어요.
```

### 표시 정보

```txt
상품명
별점 입력 유도
재구매 의사 입력 유도
후기 작성 버튼
```

### CTA

```txt
후기 작성하기
```

## 5.1.9 데이터 출처 안내 영역

### 목적

외부 쇼핑 API와 공공데이터를 사용하는 서비스이므로, 가격과 정보의 기준을 명확하게 안내한다.

### 표시 문구

```txt
상품 가격은 외부 쇼핑 API 또는 데모 데이터를 기준으로 표시됩니다.
최종 가격, 배송비, 재고 여부는 쇼핑몰에서 확인해주세요.
동물병원 정보는 공공데이터를 기반으로 제공됩니다.
```

# 5.2 상품 검색 페이지 `/products`

## 화면 목적

사용자가 반려동물 용품을 검색하고, 관심 상품을 저장하거나 외부 쇼핑몰로 이동할 수 있게 한다.

2차에서는 검색 결과를 단순 목록이 아니라 실제 가격 확인 도구처럼 보이게 개선한다.

---

## 5.2.1 구성 요소

```txt
검색창
추천 검색어
카테고리 필터
강아지 / 고양이 필터
가격 정렬
쇼핑몰 필터
검색 결과 카드
검색 결과 없음 안내
데이터 출처 안내
```

# 5.3 저장한 상품 페이지 `/saved`

## 화면 목적

2차에서는 단순 찜 목록이 아니라 가격 추적 화면으로 재구성한다.

## 5.3.2 표시 정보

```txt

가격 하락 상품 개수
가격 상승 상품 개수
전체 저장 상품 목록
저장 당시 가격
현재 확인 가격
차액
가격 상승 / 하락 상태
마지막 가격 확인 시각
후기 작성 여부
```

## 5.3.3 상품 카드 표시 정보


## 5.3.4 가격 차액 표시 정책

# 6. 가격 정보 표시 정책

## 6.1 가격 문구 원칙

현재 서비스는 외부 쇼핑 API 데이터를 기반으로 하기 때문에, 가격 표현은 보수적으로 사용한다.

사용 가능한 표현:

```txt
현재 확인 가격
최근 확인 가격
네이버 쇼핑 기준 가격
마지막 가격 확인
```

피해야 할 표현:

```txt
실시간 최저가
무조건 최저가
전체 쇼핑몰 최저가
가장 저렴한 가격
```

---

## 6.2 공통 안내 문구

가격 정보가 표시되는 화면에는 아래 문구를 노출한다.

```txt
최종 가격, 배송비, 재고 여부는 쇼핑몰에서 확인해주세요.
```

또는

```txt
표시된 가격은 최근 확인된 가격이며, 실제 구매 가격은 쇼핑몰에서 달라질 수 있습니다.
```

# 9. 내부 상품 DB 승격 흐름

## 9.1 현재 상태

현재 `products` 테이블은 존재하지만 실제 서비스 흐름에서는 거의 사용하지 않는다.

현재는 외부 쇼핑 API 결과를 `external_products`로 저장하고, 사용자가 찜하거나 리뷰한 상품을 기준으로 데이터가 쌓이는 구조이다.

---

## 9.2 2차 정책

2차에서도 `external_products → products` 승격은 본격적으로 진행하지 않는다.

아직 상품 리뷰 수와 저장 수가 충분하지 않기 때문이다.

다만, 상품 상세 페이지와 가격 기록 구조는 나중에 내부 상품으로 승격해도 유지될 수 있게 설계한다.

---

## 9.3 추후 승격 기준

아래 조건 중 하나를 만족하면 내부 상품 DB로 승격할 수 있다.

```txt
찜 5회 이상
리뷰 3개 이상
검색 빈도 높음
관리자가 직접 지정
가격 추적 대상 지정
```

# 15. 결론

2차 설계의 핵심은 기능 확장이 아니라 화면 정보 구조 정리이다.

현재 서비스는 1차 MVP 기준으로 상품 검색, 찜, 가격 기록, 동물병원 조회, 기본 리뷰 작성까지 구현되어 있다.

2차에서는 이 기능들을 사용자가 이해하기 쉬운 화면 흐름으로 재배치한다.

핵심 방향은 다음과 같다.

```txt
홈은 대시보드로 재구성한다.
상품 검색은 실제 검색 도구처럼 개선한다.
저장한 상품은 가격 추적 중심으로 보여준다.
상품 상세 페이지를 추가해 가격 기록과 후기의 중심으로 만든다.
동물병원 페이지는 공공데이터 기반 정보 페이지로 정리한다.
리뷰는 상세 구조화보다 후기 목록 표시부터 구현한다.
```

2차 완료 후 서비스는 아래 흐름을 갖게 된다.

```txt
대시보드에서 상품을 탐색한다.
상품을 검색한다.
관심 상품을 저장한다.
저장한 상품의 가격 변화를 확인한다.
상품 상세에서 가격 기록과 후기를 확인한다.
사용해본 상품에 후기를 남긴다.
필요하면 동물병원 정보를 조회한다.
```

이 구조로 가면 기존 1차 MVP를 버리지 않고, 현재 DB 구조를 유지하면서 서비스 완성도를 높일 수 있다.
