# 펫핏(PetFit) 재설계 설계서

- 작성일: 2026-06-29
- 상태: 설계 확정 (구현 계획 작성 전)
- 성격: **제품 컨셉부터 재정의**하는 전면 재설계. 기존 v0.2~v0.5 설계서의 방향성은 고정 제약으로 두지 않는다.

---

## 1. 배경과 목적

기존 `petpick-log`는 네이버 쇼핑 가격비교에서 출발해 공공데이터 기반 생활 플랫폼(v0.5)으로 확장하려던 방향이었다. 이번 재설계에서는 제품 컨셉을 백지에서 다시 정의한다.

- **프로젝트 목적**: 실제 출시/운영. 포트폴리오가 아니라 실사용자를 모으는 서비스를 목표로 한다.
- **재설계 원칙**: 차별점을 무리하게 만들지 않는다. 유저가 0인 MVP 단계에서는 **검증된 유명 플랫폼을 벤치마킹**해 익숙함으로 시작하고, 유저가 늘어난 뒤 차별화한다.

---

## 2. 제품 컨셉

### 2.1 한 줄 정의

> **펫프렌즈식 반려용품 탐색 경험 + 다나와식 가격추적**을 결합한 반려용품 서비스.
> 익숙한 커머스 UX로 둘러보고, 가격추적·그래프로 다시 찾게 만든다.

### 2.2 벤치마크 (역할 분담)

| 축 | 벤치마크 | 역할 |
|---|---|---|
| 셸 / IA / 탐색 경험 | 펫프렌즈 · 어바웃펫 (반려 전문 커머스) | 홈 큐레이션, 카테고리 브라우즈, 아이 프로필 추천, 리뷰 → 체류·전환 |
| 핵심 가치 엔진 | 다나와 · 에누리 (가격비교) | 상품 상세 가격추이 그래프, 관심상품 가격추적, 가격 중심 정렬/필터 → **재방문 동력** |

벤치마크를 섞는 이유: 펫프렌즈식 IA는 이미 유저/방문자가 있다는 전제에서 강하다. 유저가 없는 초기에는 다나와식 가격추적의 구체적 가치로 유저를 모아야 한다. 따라서 두 축을 결합한다.

### 2.3 타겟 사용자

- 주 타겟 ①: **초보 반려인** (첫 입양 1년 차) — 무엇을, 얼마에 사야 할지 안내가 필요
- 주 타겟 ②: **가격·가성비 민감형** — 가격 비교·추적이 핵심 관심사
- 확장 타겟 ③(추후): **동반 외출 보호자** — 동반 가능 장소 탐색

### 2.4 리텐션 훅

- **관심상품 가격추적 + 가격추이 그래프**가 "다시 여는 이유".
- 자동 가격 갱신(1일 2회)으로 유저가 아무것도 하지 않아도 그래프가 채워진다.
- 가격 하락 알림은 추후 단계에서 이 훅을 강화한다(MVP 제외).

## 3. IA / 페이지 구조

```
/                  홈 — 큐레이션 추천 + 내 관심상품 가격 요약 (펫프렌즈식 홈, 로그인/비로그인 분기)
/products          가격 확인 — 네이버 쇼핑 검색 + 가격순 정렬/필터 (다나와식 리스트)
/products/[id]     상품 상세 — 현재가 + 가격추이 그래프 + 리뷰 (다나와식 상세)
/saved             관심상품 — 가격 변화 추적 카드, 그래프 진입
/pets              내 아이 프로필 — 선택 입력, 추천 정밀도 ↑ (MVP 단일 프로필)
/auth              로그인 / 회원가입
─────────── 추후 ───────────
/places   동반 장소 (확장)
/guide    반려생활 가이드 (확장)
```

헤더 메뉴: **홈 · 가격 확인 · 관심상품 · 내 아이 · 로그인/로그아웃**

## 4. 핵심 사용자 흐름

```
[비로그인]
  홈 큐레이션 카테고리 클릭
  → /products 가격 검색
  → 상품 상세에서 가격추이 그래프·리뷰 확인
  → 관심상품 담기 시도 → 로그인 유도

[로그인 후]
  → (선택) 내 아이 프로필 입력 → 홈 추천이 종·나이대 맞춤으로 정밀화
  → 관심상품 담기 → /saved에서 가격 재확인/그래프 확인
  → 자동 갱신 + 재방문으로 가격 변화 추적   ← 리텐션 루프
```

핵심 원칙: **프로필은 흐름을 막지 않는다(선택 입력).** 미입력 유저는 일반 인기 추천, 입력 유저는 나이대 맞춤 추천을 받는다.

## 5. 데이터 모델

### 5.1 그대로 재사용 (기존 Supabase 스키마)

- `profiles` — 사용자 기본
- `external_products` — 네이버 쇼핑 상품 후보 (source, external_id=productId, latest_price, raw_data 등)
- `price_histories` — 가격 시점 기록 (그래프 원천)
- `saved_products` — 관심상품, `saved_price` 저장 당시 가격
- `product_reviews` — 평점·재구매의향·내용

> `animal_hospitals`는 삭제하지 않고 보류한다(추후 `/places`).

### 5.2 신규 테이블 — `pets`

선택 입력, MVP는 사용자당 단일 프로필.

```sql
create table public.pets (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  pet_type text not null,          -- DOG | CAT
  name text,
  breed text,
  birth_year integer,              -- 나이대 계산용 (생일 정밀 입력 부담 ↓)
  weight_kg numeric,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

-- MVP: 사용자당 1행
create unique index pets_user_id_key on public.pets (user_id);

-- RLS: owner-only (profiles 정책과 동일 패턴)
alter table public.pets enable row level security;
create policy "pets are owned by user"
  on public.pets for all
  to authenticated
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);
```

## 6. 큐레이션 / 추천 로직 (룰 기반)

- 추천 전용 테이블 없음. `src/lib/curation.ts`에 정적 매핑으로 시작(ML 아님).
- **품목 카테고리**(각각 네이버 검색어 매핑): 사료 · 간식 · 배변패드 · 고양이모래 · 영양제 · 위생/샴푸 · 장난감
- **프로필 미입력**: `강아지/고양이 인기 카테고리` 일반 추천
- **프로필 입력 시**: `pet_type × 나이대(유년 / 성년 / 노년, birth_year로 계산)` 기준으로 추천 품목 우선순위를 조정 (예: 노령견 → 관절 영양제 상단)
- 추후 사용 데이터/클릭 로그 기반 고도화 여지.

## 7. 가격추적 메커니즘 (다나와식, 네이버 API 기반)

### 7.1 저장

- 관심상품 담기 → `saved_products`에 `saved_price` 기록 + `external_products` upsert.

### 7.2 자동 갱신 (1일 2회)

- 스케줄러가 보안 토큰으로 배치 엔드포인트를 호출한다.
- 엔드포인트: `POST /api/cron/refresh-prices` (요청 헤더의 시크릿 토큰 검증, 미인증 차단)
- 대상: **관심상품에 담긴 distinct `external_products`만** (유저별이 아니라 상품 단위 — 가격은 공유, 호출 절약)
- 각 상품 처리: 네이버 재검색 → `productId(external_id)` 매칭 → `price_histories`에 시점 적재 + `external_products.latest_price` 갱신
- 호출량: 추적 상품 N개 × 2회/일 = 2N콜. 네이버 일일 쿼터(약 2.5만) 대비 초기 규모에선 충분.

### 7.3 수동 갱신

- 상품 상세 / 관심상품 카드의 "지금 가격 확인" 버튼으로 즉시 재조회.

### 7.4 그래프

- `price_histories` 시계열을 경량 **자체 SVG 라인차트**로 렌더(차트 라이브러리 미도입).

### 7.5 알려진 한계 (정직하게 명시)

- 네이버는 검색 API라 동일 `productId`가 사라지면 재매칭에 실패할 수 있다.
- 그 경우 "가격 확인 불가" 상태로 표시하고 기존 히스토리는 유지한다.
- 스케줄 자동 갱신은 포함하되, **가격 하락 알림(발송 채널)은 추후**.

### 7.6 스케줄러

- **채택: GitHub Actions 스케줄 워크플로**(무료). 1일 2회(예: `0 0,12 * * *` UTC = 09시·21시 KST) `/api/cron/refresh-prices` 호출.
- 대안: Vercel Cron(간단하나 Hobby 플랜은 빈도 제한으로 2회/일에 Pro 필요 가능). 무료 우선 원칙에 따라 GitHub Actions 채택.

## 8. 리뷰

- `product_reviews` 재사용. 상품 상세에서 평점·재구매의향·내용 작성/조회.

## 9. 기술 스택 및 코드 처리 방침

### 9.1 스택 유지

Next.js 16 / React 19 / TypeScript / SCSS / Supabase / Vercel. (Tailwind·CSS Module 미사용 유지)

### 9.2 재사용 — 백엔드 / 데이터 / 로직 계층에 한정

- `src/lib/naver-shopping.ts`, `src/lib/pet-search.ts`
- `src/app/api/products/search/route.ts` 및 검색·저장·가격기록·리뷰 데이터 로직
- Supabase 스키마(위 5장), auth 연동
- `src/types/product.ts`, `src/lib/format.ts` 등 유틸

### 9.3 프론트엔드 — 전면 신규 (기존 UI 전부 폐기)

> **명시: 기존 프론트엔드 화면 디자인은 전부 무시하고 처음부터 새로 설계한다.**

- 폐기 대상: 모든 페이지의 UI/마크업(`page.tsx`/`*-client.tsx`의 화면 부분), `src/components/Header.tsx`, `src/app/globals.scss`, `src/app/styles/*`(색상/타이포 토큰), 레이아웃.
- 기존 화면은 **참고도 하지 않으며**, 새 디자인 언어로 처음부터 구성한다.
- 디자인 방향: 모바일 우선의 깔끔한 커머스 톤(펫프렌즈식). 상세 비주얼/토큰은 구현 단계에서 새로 정의한다.
- 데이터 로직과 UI의 경계: 데이터 패칭/상태 로직은 재사용 가능하되, 그것을 그리는 컴포넌트·스타일은 전부 신규.

### 9.4 보류 (이번 범위 밖)

- `/hospitals`(추후 `/places`로), `/guide`.

## 10. API 설계 (요약)

| 엔드포인트 | 역할 | 비고 |
|---|---|---|
| `GET /api/products/search` | 네이버 쇼핑 검색 | 기존 유지 |

## 13. 개발 우선순위 (개요)

1. 신규 디자인 토큰/레이아웃/Header 등 프론트 기반 재구축
2. `pets` 테이블 + `/pets` 프로필(선택 입력)
3. `lib/curation.ts` 추천 매핑 + 홈 큐레이션 재구성
4. `/products` 가격 검색·정렬 UI 신규
5. `/products/[id]` 상세 + 가격추이 SVG 그래프 + 리뷰
6. `/saved` 관심상품 추적 카드
7. `/api/cron/refresh-prices` + GitHub Actions 스케줄(1일 2회)
8. 가격 표시 정책 문구 반영

세부 단계와 작업 분해는 구현 계획(plan) 문서에서 다룬다.
